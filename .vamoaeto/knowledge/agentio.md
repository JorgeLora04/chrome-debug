- **Page rendering pattern:** `'use client'` → `useModuleAccess('module')` → loading UI → `AccessRestricted` fallback → content
- **Data flow:** Convex `useQuery` (real-time) / `useMutation` (writes) / `useAction` (external APIs); ElevenLabs webhooks → Next.js API → Convex mutations → subscriptions push to frontend
- **State management:** Convex for server state (NO Redux); React Context for locale + sentiment notifications; localStorage for language persistence
- **Background jobs:** Convex crons — webhook-backfill (hourly), analytics-refresh (daily 2AM UTC), check-stuck-calls (every 5 min); Vercel Workflow via `withWorkflow()` in next.config

## Architectural Patterns From Plan Docs

### Rolling composite-key batch upsert
**Decision:** Group related webhook-triggered events into a single logical entity using a composite key like `(companyId, configurationId, externalWorkflowId, dayKey)`, backed by a dedicated Convex index for O(1) upsert lookup. First trigger creates, subsequent ones append.
**Why:** Avoids per-event record pollution (analytics noise, cluttered UI) while preserving idempotency and per-source isolation.
**Reusable when:** Any time external systems fire granular events (HubSpot, Zapier, CRM webhooks) that should collapse into a daily/hourly/sessional parent record.

### Timezone-aware dayKey with UTC cosmetic close-out
**Decision:** Store grouping `dayKey` in the tenant's timezone (company-configurable, default `America/New_York`), but run close-out crons in UTC.
**Why:** Users perceive "today" in local time; crons don't need per-tenant fanout if close-out is purely cosmetic.
**Reusable when:** Multi-tenant date-scoped grouping where display correctness matters but exact rollover timing does not.

### Direct single-call dispatch over batch-start for trigger-driven calls
**Decision:** Webhook-triggered dials call `elevenlabs.makeCall` directly (single outbound) rather than reusing `startBatchCalls`.
**Why:** Batch-start re-dispatches every pending row, causing duplicate dials on each trigger.
**Reusable when:** Reusing batch-style data models for streaming/trigger inputs.

### Phone-first contact upsert with external-ID fallback
**Decision:** Upsert sequence is `phone → externalId (hubspotId) → insert`. Phone collisions intentionally collapse records.
**Why:** Platform identity is phone-first (E.164). External IDs are secondary.
**Reusable when:** Any contact/lead sync from external CRMs.

### Custom webhook tool over native provider integration
**Decision:** Replace vendor's built-in tool (e.g., `hubspot_search_contacts`) with a company-owned webhook endpoint that shapes the response.
**Why:** Native tools return raw/default fields, don't support custom properties, aren't shaped for LLM consumption.
**Reusable when:** Vendor agent tool returns too much, too little, or wrong shape — need deterministic LLM-friendly payloads.

### Split tool response: `shareable` vs `internal_only` + `instructions_for_agent`
**Decision:** Tool responses return three buckets: fields the agent may speak aloud, context-only fields, and a plain-language rules string.
**Why:** LLM respects structure better than out-of-band prompt rules; co-locating policy with data reduces leakage.
**Reusable when:** Designing any agent-callable tool returning mixed-sensitivity data.

### Cascading conversation/agent/company identity resolution
**Decision:** Resolve `conversation_id` via cascade: call record → external id → agent record → company. Same pattern across availability, booking, contact lookup.
**Why:** Webhook tools get inconsistent identifiers depending on lifecycle timing.
**Reusable when:** Any new webhook tool handler that must identify caller/tenant context.

### Modal-to-page migration for growing feature complexity
**Decision:** When a modal exceeds dialog comfort (~600 LOC, multi-section, mobile use), migrate to dedicated `/[resource]/new` and `/[resource]/[id]` routes with section components taking `{formData, onChange, disabled}`.
**Why:** Modals have hard space ceilings and mobile-hostile padding. Routes unlock side-by-side layouts and bookmarking.
**Reusable when:** Any CRUD modal approaching ~600 LOC or needing tablet/mobile support.

### Vercel AI Gateway over direct provider SDKs
**Decision:** Use the `ai` package with Gateway model strings (`anthropic/claude-sonnet-4-5`, `openai/gpt-4o`) instead of `@ai-sdk/anthropic` + `ANTHROPIC_API_KEY`.
**Why:** Model switching is a string change; unified billing/observability; no per-provider keys.
**Reusable when:** Any new LLM-powered feature.

### FIFO-capped per-entity history table
**Decision:** For undo/restore over user-replaceable content (prompts, configs), persist `history` table keyed by `(targetId, field)` with hard cap (e.g., 5) — on insert delete oldest if at cap.
**Why:** Unbounded history bloats storage and UX. Fixed cap = predictable cost.
**Reusable when:** Any "Apply / Replace" action where users benefit from recovering recent prior versions.

### Apply-via-fenced-codeblock convention
**Decision:** AI responses that mutate form fields wrap output in typed markdown fences (` ```prompt`, ` ```first-message`); UI parses fences and renders per-target "Apply" buttons.
**Why:** Keeps model output conversational while giving UI a deterministic extract point.
**Reusable when:** Lightweight AI-assisted editors proposing replacements for known fields.

### Initial-snapshot form state to avoid sync races
**Decision:** Forms load from initial query snapshot (`useMemo` captured), not live Convex subscription, so background `syncFromProvider` can't clobber in-flight edits.
**Why:** Real-time subscriptions are great for lists, dangerous for edit buffers.
**Reusable when:** Any edit form over a table with background sync paths writing same rows.

## Compound Solutions (Hard-Won Knowledge)

### Integration Patterns (HubSpot, ElevenLabs, WhatsApp, Polar, Twilio)

- **Coerce third-party JSON at the edge** — HubSpot sends numeric IDs as JSON numbers (e.g., `215595174227.0`); always stringify before passing to Convex `v.string()` validators. _hubspot-rolling-daily-batch-trigger.md_
- **Never reuse batch-start for per-record triggers** — `startBatchCalls` re-dispatches ALL pending calls; use `elevenlabs.makeCall` directly for single-contact triggers. _hubspot-rolling-daily-batch-trigger.md_
- **Phone variant lookup is mandatory** — HubSpot stores numbers inconsistently; try `raw`, `digits-only`, `last-10`, `+digits` across `phone`, `mobilephone`, `hs_whatsapp_phone_number`. _hubspot-rolling-daily-batch-trigger.md_
- **Rolling-batch dayKey uses company timezone** — store IANA tz on companies, compute with `Intl.DateTimeFormat('en-CA', { timeZone })` (V8-safe in Convex). _hubspot-rolling-daily-batch-trigger.md_
- **HubSpot "Customize request body" is key/value only** — cannot paste raw JSON; Key auto-fills from property if blank (type key first). _hubspot-rolling-daily-batch-trigger.md_
- **HubSpot segment filters default to AND** — groups are ORed by default; verify connector text. Active segments re-evaluate every 5–15 min (not real-time). _hubspot-rolling-daily-batch-trigger.md_
- **HubSpot workflow "Test" does NOT fire webhooks** — curl end-to-end with real payloads before trusting it. _hubspot-rolling-daily-batch-trigger.md_
- **Mid-call tools can't assume `conversationId` exists on call record** — batch calls only get it at post-call webhook; use cascading fallbacks (externalId → widget → ElevenLabs API → backfill). _hubspot-booking-webhook-call-race-condition.md_
- **HubSpot booking endpoint is `/scheduler/v3/meetings/meeting-links/book`** with slug in BODY, not path. _hubspot-booking-webhook-call-race-condition.md_
- **Availability endpoint requires `encodeURIComponent()` on slug** — slugs with `/` must be encoded or API gateway 404s before reaching service. _hubspot-meeting-availability-404-wrong-endpoint.md_
- **Always `encodeURIComponent()` user-supplied path values** — cost is zero, guards against `/`, `?`, `#`, `%`. Log full URL on error. _hubspot-meeting-availability-404-wrong-endpoint.md_
- **Normalize timestamps at system boundaries** — ElevenLabs sends Unix ms strings, HubSpot expects ISO 8601; detect with regex, convert. _hubspot-booking-webhook-call-race-condition.md_
- **Response parsing must chain multiple field names** — HubSpot returns engagement IDs as `engagementId || id || calendarEventId || meetingId || contactId`. _hubspot-booking-webhook-call-race-condition.md_
- **Always gate HubSpot calls on `companies.hubspot.enabled`** and persist booking errors on the call record for debugging. _hubspot-booking-integration.md_
- **ElevenLabs batch calls return no duration/audio** — derive duration from last transcript entry's `time_in_call_secs`. _batch-completion-pipeline-fixes.md_
- **Transcripts come in multiple shapes** — normalize array `[{role, message}]`, string, and `turns` formats at the boundary. _batch-completion-pipeline-fixes.md_
- **Three-tier stuck-call timeouts** — connecting >5min → no_answer, pending >15min → no_answer, in_progress >30min → failed. _batch-completion-pipeline-fixes.md_
- **ElevenLabs tool properties must be arrays, not objects** — dynamic vars use `{{var}}`, system vars like `system__conversation_id` are separate from LLM-prompted fields. _elevenlabs-webhook-tool-configuration.md_
- **Phone numbers MUST be E.164** for ElevenLabs batch API. _batch-calling-flow.md_
- **Three-layer sync strategy for external conversation data** — webhook primary + auto-sync on page load + manual refresh. _widget-conversation-sync.md_
- **Convex runtime is V8, not Node** — no `crypto`, `fs`, `Buffer`; use Web Crypto (`crypto.subtle`), manual XOR for timing-safe compare, `TextEncoder`. _whatsapp-meta-graph-api-waba-lookup-and-convex-crypto.md_
- **Meta Graph API has no reverse edge from phone number → WABA** — `whatsapp_business_account` field doesn't exist on phone node; require manual WABA ID input. _whatsapp-meta-graph-api-waba-lookup-and-convex-crypto.md_

- **Use System User Tokens in production** — permanent, not tied to a person; validate on save with test API call; mask in UI. _whatsapp-meta-graph-api-waba-lookup-and-convex-crypto.md_
- **Never hardcode env defaults when a config table exists** — search schema for existing config (e.g., `pricingConfigs`) before using `process.env.X || fallback`. _polar-billing-sync-and-pricing-config.md_
- **External service sync pattern** — when mutating state mirrored externally, schedule an action with `ctx.scheduler.runAfter(0, ...)` since mutations can't call actions. _polar-billing-sync-and-pricing-config.md_
- **Use `internalQuery`/`internalAction` in scheduled contexts** — public queries requiring auth fail when invoked from scheduler. _polar-billing-sync-and-pricing-config.md_
- **Scope UI options to company's actual enabled modules** — never use global `COMPANY_SCOPED_MODULES`; join `companyModules` filtered by `enabled:true`. _polar-billing-sync-and-pricing-config.md_

### Runtime Validation (Convex)

- **Convex validators are STRICT** — reject any field not in validator; never spread raw objects into mutation args. _convex-validation-extra-fields.md_
- **Let mutations own their timestamps** — set `updatedAt`/`createdAt` inside the handler, never pass as arguments. _convex-validation-extra-fields.md_
- **Prefer conditional spreads for optional args** — `...(x ? {field: x} : {})` instead of building a patch object. _convex-validation-extra-fields.md_
- **Convex OCC makes check-then-insert unsafe under concurrent writes** — both mutations read same snapshot, both insert, no conflict. _whatsapp-duplicate-call-dedup-occ-race.md_
- **Post-hoc dedup via scheduled mutation** is the correct fix for webhook races — schedule cleanup 2+ seconds later so both inserts commit; make it idempotent and merge-before-delete. _whatsapp-duplicate-call-dedup-occ-race.md_
- **Post-insert dedup in the same mutation does NOT work** — each mutation only sees its own write; the other hasn't committed. _whatsapp-duplicate-call-dedup-occ-race.md_
- **Put scheduled mutations in a separate file** — avoids circular `_generated/api` dependencies that cause silent scheduler failures. _whatsapp-duplicate-call-dedup-occ-race.md_
- **Retry-with-delay for chained mutations** — auto-start batch retries up to 10× every 3s waiting for calls to be inserted. _batch-completion-pipeline-fixes.md_
- **Idempotency checks on scheduled work** — `if (batch.autoStart.triggeredAt) return` prevents duplicate execution. _batch-completion-pipeline-fixes.md_
- **Webhook handlers must extract ALL data** — don't save only `status`; also persist `durationMs`, `audioUrl`, `transcriptUrl` with existence guards. _batch-completion-pipeline-fixes.md_
- **Early-return in cron handlers breaks downstream logic** — avoid `if (x.length === 0) return` when later checks still need to run. _batch-completion-pipeline-fixes.md_
